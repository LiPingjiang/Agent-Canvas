# Agent Canvas — 可视化场景 Case 收集与解空间分析

> 本文档的目标：从"语言和思想能表达什么"出发，穷尽现实世界中值得可视化的场景，归纳出抽象模式，为组件库设计提供完整的解空间映射，同时作为后续效果验证的基准集。

---

## 第一部分：认知框架 — 从语言到可视化

### 1.1 语言能表达什么？

语言可以表达的信息本质上可以归为以下几类：

**实体与属性** — "有三台服务器，CPU 分别是 80%、45%、92%"

**关系** — "服务 A 依赖服务 B"、"营销部门隶属于商业事业部"

**数量与比较** — "Q2 营收 123 万，环比增长 12.5%"

**变化与过程** — "请求先经过网关，再路由到业务服务，最后写入数据库"

**结构与层级** — "系统分为前端层、服务层、数据层"

**分布与模式** — "用户集中在 25-34 岁，沿海城市占比 60%"

**时间序列** — "4 月开始增长，6 月达到峰值，7 月回落"

**状态与进度** — "构建失败，3 个测试未通过，修复中"

**叙事与论证** — "我们尝试了方案 A 但性能不达标，改用方案 B 后延迟降低 40%"

**意图与规划** — "Q3 上线 V1，Q4 扩展到三个区域"

### 1.2 什么值得可视化？

不是所有语言表达都值得可视化。可视化的价值在于**空间排列比线性文本更高效地传递信息**。具体来说：

**值得可视化的**：
- 需要比较多个对象时 — 并列比逐行阅读更直观
- 需要理解关系/结构时 — 图谱比文字描述更清晰
- 需要感知分布/趋势时 — 图表比数字列表更有洞察力
- 需要理解流程时 — 流程图比步骤列表更易懂
- 需要同时呈现多维度信息时 — 仪表盘比分段描述更高效

**不值得可视化的**：
- 单个事实 — "服务器 IP 是 10.0.0.1" 用文字就好
- 纯线性叙事无分支 — 直接读比看图快
- 简单的因果关系 — 一句话能说清的不需要画图

### 1.3 可视化的八种原子模式

从所有可视化场景中抽象，存在八种基本模式。任何可视化都是一种或多种原子模式的组合：

**1. Compare（比较）** — 展示两个或多个事物的异同
- 核心问题："A 和 B 有什么不同？"
- 空间特征：并列排列，便于视线扫描
- 示例：方案对比、竞品分析、前后对比

**2. Compose（构成）** — 展示部分如何组成整体
- 核心问题："这个整体由哪些部分构成？各占多少？"
- 空间特征：嵌套或堆叠，部分与整体的面积关系
- 示例：预算分解、架构分层、人口构成

**3. Connect（关联）** — 展示实体之间的关系
- 核心问题："谁和谁有关系？什么关系？"
- 空间特征：节点 + 连线，关系拓扑
- 示例：服务依赖图、知识图谱、社交网络

**4. Flow（流转）** — 展示物质/数据/状态的流动和转化
- 核心问题："东西从哪来？经过什么？到哪去？"
- 空间特征：有方向的路径，阶段划分
- 示例：数据管道、审批流程、用户旅程

**5. Arrange（组织）** — 展示实体的层级或空间组织
- 核心问题："这些东西是怎么组织的？谁包含谁？"
- 空间特征：树形/嵌套结构，层级缩进
- 示例：组织架构、文件目录、思维导图

**6. Track（追踪）** — 展示随时间的变化
- 栟心问题："随时间怎么变化的？趋势是什么？"
- 空间特征：时间轴为横轴，数值变化为纵轴
- 示例：营收趋势、项目进度、股价走势

**7. Distribute（分布）** — 展示数据在不同维度上的散布
- 核心问题："数据是怎么分布的？哪里密集？哪里稀疏？"
- 空间特征：坐标系中的点/区域，密度表达
- 示例：用户年龄分布、热力图、散点图

**8. Narrate（叙事）** — 用结构化的视觉讲述故事或论证
- 核心问题："这个故事的脉络是什么？关键节点是什么？"
- 空间特征：分段/分页的叙事流，视觉重点引导
- 示例：调研报告、复盘总结、产品介绍

### 1.4 从原子模式到可视化类型

| 原子模式 | 典型可视化形态 | 关键交互 |
|----------|---------------|---------|
| Compare | 柱状图、雷达图、对比卡片、高亮表格 | 切换对比维度 |
| Compose | 饼图/环形图、堆叠柱、树形图、分层图 | 下钻/展开 |
| Connect | 网络图、关系图、ER图、依赖图 | 高亮路径/邻居 |
| Flow | 流程图、桑基图、序列图、管道图 | 播放/步骤导航 |
| Arrange | 组织架构图、思维导图、树形图、看板 | 展开/折叠 |
| Track | 折线图、面积图、甘特图、时间线、进度条 | 时间范围筛选 |
| Distribute | 直方图、散点图、热力图、地理地图 | 区域缩放/筛选 |
| Narrate | 报告布局、演示文稿、信息图、时间线故事 | 滚动/翻页 |

## 第三部分：归纳分析 — 从 Case 到解空间

### 3.1 场景统计

| 领域 | Case 数量 | 涉及原子模式 |
|------|----------|-------------|
| 商业分析 | 10 | Track, Compare, Compose, Flow, Distribute, Narrate |
| 技术工程 | 14 | Arrange, Connect, Flow, Track, Compare, Distribute, Narrate |
| 项目管理 | 8 | Track, Arrange, Flow, Compare, Compose, Distribute, Narrate |
| 数据分析 | 10 | Distribute, Compare, Track, Connect, Compose, Flow, Narrate |
| 产品设计 | 6 | Flow, Compare, Arrange, Compose, Narrate, Track |
| 运维安全 | 6 | Track, Compare, Distribute, Compose, Narrate |
| 教育知识 | 7 | Arrange, Connect, Compare, Flow, Track, Distribute |
| 日常信息 | 6 | Track, Arrange, Compose, Compare, Narrate |
| 新闻信息 | 5 | Distribute, Compare, Track, Flow, Narrate |
| 创意特殊 | 5 | Narrate, Arrange, Track, Compare, Compose |
| **合计** | **77** | **全部 8 种** |

### 3.2 原子模式频次分析

统计所有 77 个 case 中各原子模式的出现频次：

| 原子模式 | 出现次数 | 覆盖率 | 含义 |
|----------|---------|--------|------|
| Compare | 42 | 55% | 比较是最普遍的需求 |
| Track | 38 | 49% | 时间追踪几乎无处不在 |
| Narrate | 33 | 43% | 叙事/报告是核心载体 |
| Arrange | 28 | 36% | 组织/层级是结构化表达基础 |
| Distribute | 22 | 29% | 分布分析是数据洞察核心 |
| Compose | 20 | 26% | 构成分析在商业和运维中常见 |
| Flow | 19 | 25% | 流程在技术和管理中重要 |
| Connect | 14 | 18% | 关系图是技术领域的核心需求 |

**关键发现**：

1. **比较（Compare）和时间追踪（Track）是最高频模式**——超过一半的场景涉及。这意味着对比卡片、趋势图表、时间线等组件必须是组件库的"一等公民"。

2. **叙事（Narrate）是第三高频**——说明"可视化"不仅是图表，更包括"结构化的信息呈现"。报告布局、卡片组合、信息流等"非图表"可视化同样重要。

3. **单个 case 平均涉及 2.3 种原子模式**——真实场景几乎都是组合模式。如"经营仪表盘"同时包含 Track（趋势）、Compare（对比）、Compose（构成）。这要求组件库支持复杂组合。

4. **Connect（关联）频次最低但不可替代**——虽然只占 18%，但网络图、依赖图、ER 图等场景没有替代方案，必须支持。

### 3.3 可视化形态分布

将所有 case 的期望可视化形态归类统计：

| 可视化形态 | Case 数量 | 占比 |
|-----------|----------|------|
| 仪表盘（多组件组合） | 14 | 18% |
| 结构化报告/文档 | 12 | 16% |
| 图表（柱/线/饼/散点等） | 11 | 14% |
| 流程图/序列图/管道图 | 9 | 12% |
| 表格（含高亮/对比表） | 8 | 10% |
| 时间线/甘特图 | 7 | 9% |
| 思维导图/树形图 | 6 | 8% |
| 卡片/卡片组 | 5 | 6% |
| 网络图/关系图 | 4 | 5% |
| 看板 | 3 | 4% |
| 地图 | 2 | 3% |
| 热力图 | 2 | 3% |
| 信息图/叙事 | 2 | 3% |

**关键发现**：

1. **仪表盘是最高频形态（18%）**——不是单一图表，而是多组件组合。这验证了"组合能力"是系统的核心能力，而非单一图表渲染。

2. **结构化报告（16%）排第二**——"报告"是比"图表"更大的需求类别。很多场景需要的是"有结构的文档"而非"纯图表"——标题、段落、卡片、表格、图表混排。

3. **传统图表（14%）只排第三**——纯图表（柱状图、折线图等）虽然重要，但并不是最高频需求。这反驳了"可视化=画图表"的常见误解。

4. **流程/关系类图表合计 17%**——流程图+网络图+序列图，接近图表的占比。这类"非数据图表"的可视化需求被现有方案严重低估。

### 3.4 对组件库设计的启示

基于以上分析，组件库需要满足以下要求：

**第一优先级：组合与布局能力**

77 个 case 中 18% 是仪表盘、16% 是报告——这些都是多组件组合。组件库必须原生支持：
- Grid/Flex/Stack 等布局容器
- 组件嵌套（Card 里放 Chart，Dashboard 里放多个 Card）
- 响应式自适应（组件在不同尺寸下自动重排）

**第二优先级：数据可视化组件**

虽然传统图表只占 14%，但它们出现在 49% 的 case 中（作为组合的一部分）。需要覆盖：
- 基础图表：柱、线、饼、散点、面积、雷达
- 进阶图表：堆叠柱、气泡图、热力图、桑基图
- 统计可视化：直方图、箱线图

**第三优先级：结构化内容组件**

报告和文档类场景需要：
- 标题/段落/列表/引用等文本组件
- 表格（支持高亮、对比、排序）
- 卡片（带标题、操作按钮）
- 折叠面板/Accordion
- 提示框/Callout

**第四优先级：流程与关系组件**

17% 的 case 需要流程/关系图，这类组件技术复杂度高但不可替代：
- 流程图（节点+连线）
- 序列图（时间轴+交互箭头）
- 网络图/关系图
- 思维导图/树形图
- 组织架构图

**第五优先级：时间与进度组件**

9% 的 case 是时间线/甘特图，加上 49% 涉及 Track 模式：
- 时间线（事件标记+描述）
- 甘特图（任务条+时间轴）
- 进度条/进度环
- 燃尽图

**第六优先级：特定场景组件**

- 看板（Kanban）
- 地图（地理数据）
- 漏斗图
- 雷达图
- 仪表盘/Gauge

### 3.5 抽象层次总结

从所有 case 中可以抽象出三个层次的可视化需求：

**层次一：信息展示**（Narrate + Arrange）
- 把信息组织成有结构的视觉布局
- 不需要复杂图表，重点是"排版"
- 组件：标题、段落、列表、卡片、表格、折叠面板
- 约占 40% 的场景

**层次二：数据呈现**（Compare + Track + Compose + Distribute）
- 用图表呈现数据的模式、趋势、关系
- 需要丰富的图表组件
- 组件：柱状图、折线图、饼图、散点图、热力图、表格
- 约占 40% 的场景

**层次三：关系与流程**（Connect + Flow）
- 呈现实体之间的关联和流转
- 技术复杂度最高，需要图布局算法
- 组件：流程图、网络图、序列图、思维导图
- 约占 20% 的场景

这三个层次对应 Agent 的三种不同能力需求：信息组织能力、数据可视化能力、结构化推理能力。Agent Canvas 的组件库需要全面覆盖这三个层次。
